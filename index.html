Testing on a distributed ledger is painful

Background

Hedera vs. Hiero

Hedera

A public distributed-ledger network. Uses Hashgraph consensus (gossip-about-gossip + virtual voting), not a traditional blockchain. Native coin: HBAR.

Hiero

The open-source codebase that powers Hedera, contributed to the Linux Foundation for community governance.

Rule of thumb: Hiero = the software  •  Hedera = the public network that runs it.

You'll see "Hiero consensus node" and "Hedera consensus node" — treat them as the same software.

The core value

Why run a network locally?

Control

Reset to genesis, fund accounts with fake HBAR, full admin.

Isolation

No shared testnet, no rate limits, works offline.

Disposable

Spin up, test, tear down — repeat on every change.

If Hedera's public network is grid electricity, Solo is a generator in your garage — and the action is a robot that switches it on automatically.

What gets deployed

A Hiero network = several services

Consensus Node

Orders & executes transactions, holds ledger state. The source of truth.

Mirror Node

Stores history; serves REST / gRPC / web3 query APIs.

JSON-RPC Relay

Ethereum-compatibility layer (MetaMask, Hardhat, ethers).

Explorer

Web UI to browse accounts & transactions in a browser.

Block Node (opt.)

Stores & distributes the block-stream output.

HAProxy / Envoy

Proxies in front of the node (native gRPC & gRPC-Web).

Solo's whole job: install and wire these together for you.

Components in depth (1/2)

The write side & the read side

Consensus Node — the writer

  • Receives transactions, runs Hashgraph to agree on order
  • Executes them & maintains authoritative state
  • Java software from Hiero — its version = hieroVersion / --release-tag
  • Local Solo network: usually 1 node (2 in "dual mode")

Mirror Node — the reader

  • Consensus nodes are built to agree, not to answer questions
  • Mirror node subscribes to finalized data, stores it
  • Exposes friendly REST / gRPC / web3 APIs
  • "What hit this account?" → ask the mirror node
Components in depth (2/2)

Compatibility & plumbing

JSON-RPC Relay

Translates Ethereum JSON-RPC (eth_chainId, eth_sendRawTransaction) into Hedera ops — so EVM tooling "just works." Needs the mirror node.

Explorer

A web UI (like Etherscan, but local) at localhost:38080 to inspect accounts, txns, balances, tokens.

HAProxy

Load balancer / TCP proxy in front of each node's gRPC port (50211). Native Hedera SDKs connect through it.

Envoy

Modern proxy providing gRPC-Web, so browsers (which can't speak raw gRPC) can reach the node.

The tool itself

What Solo is

"An opinionated CLI tool to deploy and manage standalone test networks."

It orchestrates consensus nodes, mirror nodes, explorers and relays via Helm charts on Kind clusters.

How it runs

The dependency stack

Docker→ Kind→ Kubernetes→ Helm→ Hiero components

Docker

Container engine — everything runs inside it.

Kind

Kubernetes-IN-Docker: a throwaway K8s cluster.

Kubernetes

Manages the containers (pods, services, namespaces).

Helm

The "package manager for K8s" — installs each component as a chart.

Hiero

The actual consensus node, mirror node, relay, explorer.

Resources

≥ 12 GB RAM, ≥ 6 CPU cores (16/8 for multi-node).

Using Solo — the easy path

One-Shot: a network in one command

solo one-shot single deploy

This single command:

  • Creates / connects a Kind cluster
  • Deploys all components
  • Sets up & funds default accounts
  • Port-forwards the endpoints to localhost

Variants:

solo one-shot multi deploy \
  --num-consensus-nodes 3

solo one-shot single destroy

Ideal for demos & the Quickstart. First run is slower (image pulls); later runs reuse the cache.

Using Solo — the controlled path

Manual deploy: what one-shot hides

solo init --dev
solo cluster-ref config connect --cluster-ref kind-solo-e2e --context kind-solo-e2e
solo deployment config create -n solo --deployment solo-deployment
solo deployment cluster attach --deployment solo-deployment --cluster-ref kind-solo-e2e --num-consensus-nodes 1
solo keys consensus generate --gossip-keys --tls-keys -i node1 --deployment solo-deployment
solo cluster-ref config setup -s solo-cluster
solo consensus network deploy -i node1 --deployment solo-deployment --release-tag v0.72.0
solo consensus node setup   -i node1 --deployment solo-deployment --release-tag v0.72.0
solo consensus node start   -i node1 --deployment solo-deployment

This is exactly what the GitHub Action runs under the hood. Command grammar changed at v0.44.0 (newer = verb-grouped: consensus network deploy).

The mental model

cluster-ref • namespace • deployment

cluster-ref

A named pointer to a Kubernetes cluster + context.
"Which cluster?"

namespace

A virtual partition inside a cluster that isolates resources.
"Which slice?"

deployment

Solo's named network tying cluster + namespace + nodes + keys.
"Which network?"

"Create deployment solo-deployment, in namespace solo, on cluster-ref kind-solo-e2e, with 1 node."

Cryptography

Keys & accounts

Keys Solo generates

  • Gossip keys — authenticate node-to-node gossip
  • TLS keys — encrypt gRPC traffic in transit
  • TSS — threshold signatures (network-level)

Account key types

ED25519ECDSA
Hedera defaultsecp256k1 (BTC/ETH)
Fast & simpleEVM / MetaMask compatible

The action creates both account types (funded with 10,000,000 HBAR). Private keys are read from a Kubernetes Secret (base64-decoded).

hiero-solo-action

The same network — inside CI

A composite GitHub Action: an ordered list of shell steps in action.yml. Drop it into any workflow:

  • No manual Java / Node / Kind / Helm / Solo setup
  • One step → a live, throwaway network
  • Hands your tests funded account credentials
- name: Setup Hiero Solo
  uses: hiero-ledger/hiero-solo-action@v0.8
  id: solo

- name: Use Hiero Solo
  run: |
    echo "Account: ${{ steps.solo.outputs.accountId }}"
    echo "Key:     ${{ steps.solo.outputs.privateKey }}"
Inside action.yml

What the action does, in order

① Prepare

Validate Node ≥ 22 · install Java 21, Node, Python, wget · install Kind & Helm · clean prior runs

② Install & detect

npm-install the right Solo package · detect version (solo-ge-0440) to pick command grammar

③ Deploy network

Full manual sequence → then kubectl port-forward services to localhost

④ Mirror node

If installMirrorNode/installRelay · forwards REST/gRPC/web3/Java APIs

⑤ Relay

If installRelay · adds JSON-RPC relay · forwards port 37546

⑥ Accounts

Create & fund ECDSA + ED25519 · publish as outputs

A bundled extractAccountAsJson.py regex-parses the account JSON from Solo's text output.
